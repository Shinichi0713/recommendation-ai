
## Use Case

ユースケース（Use Case）を定義することは、家を建てる前に「その家で誰が、どんな生活を送るのか」を詳しく書き出す作業に似ています。

単に「何を作るか（機能）」ではなく、 **「誰が、何のために、どう使うか」** を明確にするために存在します。具体的には、以下の4つの大きな役割があります。


### 1. ユーザー視点での「価値」を定義するため

システムができること（機能一覧）を並べるだけでは、「それが本当にユーザーの役に立つのか」が見えにくいことがあります。
ユースケースを定義することで、 **「ユーザーが特定の目的（ゴール）を達成するまでの道のり」** が可視化され、無駄な機能の開発を防ぎ、本当に必要な価値に集中できます。

### 2. 開発チームとステークホルダーの「共通言語」にするため

エンジニア、デザイナー、クライアント、営業担当者は、それぞれ専門用語や視点が異なります。
「ログイン機能」という言葉一つとっても、人によって想像する挙動はバラバラです。ユースケースとして「ユーザーがIDを忘れ、再発行してログインする」といった具体的なシナリオを共有することで、 **認識のズレ（手戻り）** を最小限に抑えられます。

### 3. テストの「合格基準」にするため

「アプリケーションが完成した」と判断する基準は、単にコードが動くことではありません。
ユースケースがあれば、 **「定義されたシナリオ通りにユーザーが目的を達成できるか」** という視点でテストケース（受入テスト）を作成できます。これがそのまま、品質保証のガイドラインになります。

### 4. システムの「境界」を明確にするため

どこまでをシステムが自動で行い、どこからを人間（アクター）が行うのかという境界線を引くことができます。これにより、開発のスコープ（範囲）が明確になり、「あれもこれも」と機能が膨れ上がる「スコープクリープ」を防ぐことができます。


### ユースケースを構成する主な要素

検討を始める際は、以下の3つの要素を意識すると定義しやすくなります。

| 要素 | 説明 |
| --- | --- |
| **アクター (Actor)** | システムを利用する人や、連携する外部システム（「管理者」「一般ユーザー」など）。 |
| **シナリオ (Scenario)** | アクターとシステムのやり取りの具体的な手順（「メインフロー」や「エラーフロー」）。 |
| **ゴール (Goal)** | アクターがその操作を通じて最終的に得たい結果（「商品の注文を完了する」など）。 |

##

ユースケースの書き方に厳密な「正解」はありませんが、一般的には**「概要レベル」から「詳細手順」へと具体化していくステップ**を踏むと、漏れがなく、チーム全員が理解しやすいものになります。

以下の4つのステップで構成するのが標準的です。

---

### ステップ1：ユースケース図で全体像を描く

まずは細かな手順を追う前に、システムの「境界」と「登場人物（アクター）」、そして「主要な機能（ユースケース）」を俯瞰します。

* **アクター**: システムの外にいて、システムとやり取りをする人や外部システム。
* **ユースケース**: アクターの「目的（ゴール）」。

---

### ステップ2：基本項目を埋める

各ユースケースについて、以下の基本情報を定義します。

* **ユースケース名**: 「〜する」という動詞で終わる簡潔な名称（例：商品を注文する）。
* **アクター**: その操作を始める主役。
* **事前条件**: そのユースケースを始める前に満たされているべき状態（例：ログイン済みであること）。
* **事後条件（成功時）**: 完了したときに達成されているべき状態（例：注文がデータベースに保存され、受付メールが送信されていること）。

---

### ステップ3：メインシナリオ（ハッピーパス）を書く

何も問題が起きなかった場合の、アクターとシステムのやり取りを箇条書きで書きます。
**「アクターが〜する」「システムが〜を表示する」**という主語を明確にするのがコツです。

> **例：商品を注文する**
> 1. ユーザーが「注文確定」ボタンを押す。
> 2. システムが在庫を確認する。
> 3. システムが決済処理を行う。
> 4. システムが注文完了画面を表示する。
> 
> 

---

### ステップ4：例外・代替シナリオを追加する

メインシナリオの途中で、エラーや別のパターンが発生した場合の動きを書きます。ここが最も重要で、開発時のバグや考慮漏れを防ぎます。

* **代替シナリオ**: 別の選択肢（例：クーポンを利用して注文する）。
* **例外シナリオ**: エラー発生（例：クレジットカードが期限切れだった場合、在庫が不足していた場合）。

| 項目 | 内容 |

| **例外シナリオ** | **3a. 経路が塞がれている場合**:<br>

### 💡 良いユースケースを書くための3つのヒント

1. **「どのように（How）」ではなく「何を（What）」書く**:
「SQLを発行する」などの内部実装ではなく、「注文情報を保存する」という動作の目的を書きます。
2. **アクターの視点で書く**:
システム内部の処理ばかり書かず、「ユーザーが何を見て、何をするのか」を意識します。
3. **欲張りすぎない**:
1つのユースケースに盛り込みすぎず、大きなものは分割します。

## メリット

ユースケース（Use Case）図は、開発者にとって単なる「設計図」以上の価値を持ちます。複雑なシステムを構築するエンジニアが、これがあることで得られる具体的なメリットは主に以下の4点です。

---

### 1. 「何を作るべきか」の境界線が明確になる

開発者が最も困るのは、開発途中で「あれもこれも」と機能が膨らんでいくこと（スコープクリープ）です。

* **メリット**: システムの「中（開発範囲）」と「外（ユーザーや外部システム）」が視覚的に区別されるため、実装すべき責任範囲がひと目でわかります。
* **エンジニアの視点**: 「このボタンの裏側の処理は自分たちの担当か、外部APIの担当か」といった迷いがなくなります。

### 2. インターフェース（I/F）の設計漏れを防げる

アクター（ユーザーや外部システム）とユースケースを繋ぐ線は、そのまま「データの出入り口」を意味します。

* **メリット**: どんなアクターがシステムとやり取りするかが列挙されているため、必要なAPIエンドポイントやUI画面の出し入れを、開発の初期段階で網羅できます。
* **エンジニアの視点**: 「あ、管理者専用の管理画面も必要だったのか！」という後出しの要求による大幅な修正（手戻り）を防げます。

### 3. テスト項目（受入条件）のベースになる

ユースケース図は「ユーザーが達成したいゴール」を整理したものです。

* **メリット**: ユースケースの名称（例：「荷物を運搬する」「決済を承認する」）がそのままテストのタイトルになります。
* **エンジニアの視点**: 単なる「関数が動くか」というユニットテストだけでなく、「ユーザーが目的を達成できるか」という結合テストやE2Eテストのシナリオを、図から自動的に導き出せます。

### 4. チーム内・非技術者との「誤解」が激減する

ソースコードや複雑なクラス図は非技術者（クライアントや営業）には理解できませんが、ユースケース図なら理解可能です。

* **メリット**: 仕様のすり合わせで「言った・言わない」のトラブルが減ります。
* **エンジニアの視点**: 仕様変更の際、「この図のこの部分が変わるなら、影響範囲はここですね」と、共通の図を指差しながら具体的な工数見積もりの話をスムーズに進められます。

### 💡 開発現場でのリアルな活用例

例えば、先ほどの**e-puck**のプロジェクトであれば、以下のようなユースケース図が考えられます。

| 項目 | 開発者にとっての恩恵 |
| --- | --- |
| **アクター: ユーザー** | 遠隔操作用のUIやコマンド受付機能が必要だとわかる。 |
| **アクター: 外部センサー** | データのパース処理や通信プロトコルの実装が必要だとわかる。 |
| **ユースケース: 異常検知** | 例外処理（エラーハンドリング）のロジックを最優先で作るべきだと判断できる。 |
