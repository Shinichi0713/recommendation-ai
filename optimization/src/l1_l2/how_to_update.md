結論から言うと、**「L1+L2 の最適化で必ず L2 から最適化を進めなければならない」というわけではありません**。  
多くの実装（例：glmnet, scikit-learn）では、L1 と L2 を**同時に**最適化する形で解いています。

一方で、「L2 を先に最適化する」というのは、**naive Elastic Net**と呼ばれる 2 段階手順を指していることが多いです。その背景と、なぜ L1 だけを先に最適化するのが好ましくない場合があるかを順に説明します。

## 1. Elastic Net の目的関数と標準的な最適化

Elastic Net の目的関数は

\[
\hat{\beta} = \arg\min_{\beta} \|y - X\beta\|^2 + \lambda_2 \|\beta\|_2^2 + \lambda_1 \|\beta\|_1
\]

です[Wikipedia](https://en.wikipedia.org/wiki/Elastic_net_regularization)。

多くのライブラリ（R の glmnet, scikit-learn など）では、**座標降下法（coordinate descent）**を使って、この目的関数を**一度に**最小化します。  
このとき、L2 項は微分可能な「滑らかな部分」に含まれ、L1 項は非微分な「絶対値部分」として扱われますが、**L2 を先に最適化してから L1 を最適化する、という分離した手順にはなっていません**[UBC CS notes](https://www.cs.ubc.ca/~schmidtm/Courses/540-W16/L5.pdf)。


## 2. 「L2 から最適化を進める」のは naive Elastic Net の話

「L2 から最適化を進める」という表現は、多くの場合、**naive Elastic Net** の 2 段階手順を指しています[Zou & Hastie (2005)](https://hastie.su.domains/Papers/elasticnet.pdf)。

naive Elastic Net は次の 2 ステップで解きます：

1. **Step 1（L2 の最適化）**  
   各固定の \(\lambda_2\) に対して、Ridge 回帰（L2 正則化付き最小二乗）を解く：
   \[
   \tilde{\beta}(\lambda_2) = \arg\min_{\beta} \|y - X\beta\|^2 + \lambda_2 \|\beta\|_2^2
   \]

2. **Step 2（L1 の縮小）**  
   得られた \(\tilde{\beta}(\lambda_2)\) に対して、Lasso 型の縮小（L1 正則化）をかける：
   \[
   \hat{\beta} = \arg\min_{\beta} \|y - X\beta\|^2 + \lambda_1 \|\beta\|_1
   \]
   ただし、このときの解は Lasso の解パスに沿って求めます。

このように、**「まず L2 で安定化してから L1 でスパース化する」** という 2 段階手順が「L2 から最適化を進める」というイメージに対応します。


## 3. なぜ L1 だけを先に最適化するのは好ましくないことが多いか

### (1) L1 だけだと「相関の強い変数」の扱いが不安定

Lasso（L1 のみ）は、相関の高い特徴量が複数あるとき、そのうちの**どれか 1 つだけ**を選びがちです。  
これは L1 制約領域が「角のある多面体」であることに起因し、数値的にも不安定になりやすいです。

これに対し、L2 を入れると目的関数が**強凸（strongly convex）**になり、解が一意になり、相関の高い変数同士の係数が似た値になりやすくなります（**grouping effect**）[UBC CS notes](https://www.cs.ubc.ca/~schmidtm/Courses/540-W16/L5.pdf)。

したがって、**L1 だけを先に最適化してしまうと、相関変数の扱いが不安定になる**という問題があります。

### (2) L1 だけだと縮小が強すぎる場合がある

L1 正則化は係数を 0 に押し込む力が強いため、  
- 特徴量数がサンプル数よりはるかに多い（\(p \gg n\)）場合  
- 特徴量間に強い相関がある場合  

には、**必要な変数まで消してしまう**ことがあります。

L2 を先に（あるいは同時に）入れることで、  
- 係数の大きさを適度に抑えつつ  
- L1 によるスパース化を補助する  

という役割があります。

### (3) 数値的安定性と収束性

L1 だけの目的関数は、0 付近で微分不可能な「角」を持ち、最適化アルゴリズム（勾配法・座標降下法）の収束が遅くなったり、数値的に不安定になることがあります。

L2 項を加えると、目的関数全体が滑らかさを増し、**強凸性**が保証されるため、  
- 解が一意になる  
- 最適化アルゴリズムの収束が安定する  

という利点があります[UBC CS notes](https://www.cs.ubc.ca/~schmidtm/Courses/540-W16/L5.pdf)。

## 4. では「L1 を先に最適化」は絶対にダメか？

**必ずしもダメではありません**が、実用上は以下の理由から推奨されにくいです。

- L1 だけを先に最適化すると、上記のように相関変数の扱いや縮小の強さの面で問題が出やすい。
- Elastic Net の本来の目的は「L1 のスパース性」と「L2 の安定性・グルーピング効果」を**同時に**活かすことなので、L1 だけを先に最適化してしまうと、L2 のメリットを十分に活かせない。

そのため、多くの実装では  
- naive Elastic Net のように「L2 → L1」の 2 段階で解くか  
- あるいは L1 と L2 を**同時に**最適化する座標降下法を使う  

という設計になっています。


## まとめ

- **標準的な Elastic Net の最適化では、L1 と L2 を同時に最適化する**のが一般的です[Wikipedia](https://en.wikipedia.org/wiki/Elastic_net_regularization)。
- 「L2 から最適化を進める」というのは、**naive Elastic Net** の 2 段階手順（Ridge → Lasso 型縮小）を指すことが多く、これは相関変数の安定化や数値的安定性の観点から有用です[Zou & Hastie (2005)](https://hastie.su.domains/Papers/elasticnet.pdf)。
- **L1 だけを先に最適化することも理論的には可能**ですが、相関変数の扱いや縮小の強さの面で問題が出やすく、Elastic Net の設計思想（L1 と L2 の両方の利点を活かす）からは外れがちです。

したがって、「L2 から最適化を進める必要がある」というよりは、  
**「L2 を先に（あるいは同時に）扱うことで、L1 だけでは得られない安定性・グルーピング効果を確保する」**  
と理解するのが適切です。
